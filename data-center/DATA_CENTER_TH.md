เวอร์ชันภาษาอังกฤษ (ละเอียดกว่า): [DATA_CENTER_EN.md](DATA_CENTER_EN.md)

## 1. Data Center คืออะไร?

ลองนึกภาพโกดังขนาดใหญ่ที่ปลอดภัยสุด ๆ แต่แทนที่จะเอาไว้เก็บของ มันกลับเต็มไปด้วย **คอมพิวเตอร์** เป็นพัน ๆ เครื่อง

คอมพิวเตอร์เหล่านี้ (เรียกว่า **Server**) ทำหน้าที่รันเว็บไซต์ แอปพลิเคชัน อีเมล และระบบออนไลน์ต่าง ๆ ที่เราใช้กันทุกวัน Data Center ก็คือตัว **อาคารสถานที่** ที่ถูกสร้างขึ้นมาเพื่อดูแลให้ Server เหล่านี้ทำงานได้อย่างปลอดภัยตลอด 24 ชั่วโมง 365 วัน

### ทำไมเอาไปตั้งใน Office ธรรมดาไม่ได้?

Server ต้องการสภาพแวดล้อมที่เป๊ะมาก ๆ เพื่อให้ทำงานได้ปกติ:

- **Constant, reliable electricity (ระบบไฟฟ้าที่เสถียร)** — แค่ไฟตก 1 วินาทีก็อาจทำให้ critical system ล่มได้
- **Precise temperature and humidity (อุณหภูมิและความชื้นที่พอเหมาะ)** — Server ปล่อยความร้อน (heat) ออกมาเยอะมาก และถ้ามัน overheat เครื่องก็จะพัง
- **Physical security (ความปลอดภัยทางกายภาพ)** — เครื่องพวกนี้เก็บข้อมูลสำคัญ ถ้ามีคนนอกเข้าถึงได้คงเป็นเรื่องใหญ่
- **Fast, reliable network (เครือข่ายที่เร็วและนิ่ง)** — เพื่อให้ผู้ใช้จากทั่วโลกเชื่อมต่อเข้ามาได้ทันที

Office ทั่วไปไม่สามารถรับประกันเรื่องพวกนี้ได้ Data Center จึงเป็น **purpose-built environment** (สถานที่ที่สร้างมาเพื่อจุดประสงค์นี้โดยเฉพาะ) เพื่อให้ Server ทำงานได้ตลอดเวลา

> **เปรียบเทียบง่าย ๆ:** Data Center สำหรับ Server ก็เหมือนโรงพยาบาลสำหรับคนไข้ — เป็นสภาพแวดล้อมที่มีการควบคุมอย่างดี (ทั้งระบบ Power, Cooling, Monitoring, และ Security) เพื่อให้ทุกอย่างราบรื่น

### Data Center มีอะไรให้บ้าง?

| ความต้องการ | สิ่งที่ Data Center จัดเตรียมให้ |
|------|-------------------------------|
| ที่อยู่ปลอดภัยสำหรับอุปกรณ์ | ห้องที่ปลอดภัย มี **Racks**, ล็อกประตู และกล้องวงจรปิด |
| ไฟฟ้าที่ไม่เคยดับ | Backup generators, ระบบแบตเตอรี่, และแหล่งจ่ายไฟหลายเส้นทาง |
| การคุมอุณหภูมิ | ระบบ Air conditioning ระดับอุตสาหกรรมที่รันตลอด 24/7 |
| Network connectivity | การเชื่อมต่ออินเทอร์เน็ตความเร็วสูงออกสู่โลกภายนอก |
| Monitoring | มีคนและซอฟต์แวร์คอยมอนิเตอร์ระบบตลอดเวลา |

## 2. Data Center Standards

ก่อนจะสร้าง Data Center ต้องมี **Standards (มาตรฐาน)** ซึ่งเป็นไกด์ไลน์ที่เขียนโดยผู้เชี่ยวชาญ เพื่อให้มั่นใจว่า Data Center ถูกสร้างอย่างปลอดภัยและไว้ใจได้

เหมือนการสร้างบ้านที่ต้องมี Building code (กฎหมายควบคุมอาคาร) เรื่องการเดินสายไฟ ท่อประปา Data Center ก็มีกฎพวกนี้เหมือนกัน แต่เข้มงวดกว่ามาก

### 3 องค์กร Standards ที่สำคัญ

#### TIA-942 — The Master Blueprint
**TIA** (Telecommunications Industry Association) มี Standard ชื่อ **TIA-942** ซึ่งเปรียบเหมือน **พิมพ์เขียวหลัก** ในการออกแบบ Data Center ครอบคลุมเรื่อง:

- **Room layout and spaces** — วาง Server ตรงไหน, Cooling unit อยู่ไหน, จัด Floor Plan ยังไง
- **Cabling** — วิธีเดินสาย Network และ Power
- **Electrical power design** — การออกแบบระบบไฟให้เสถียรที่สุด
- **Cooling systems** — ระบบทำความเย็น
- **Fire protection** — การตรวจจับและดับไฟโดยไม่ทำให้อุปกรณ์พัง
- **The Tier system** — ระบบจัดระดับ (Tier I - IV) เพื่อบอกความน่าเชื่อถือ (มีอธิบายต่อด้านล่าง)

#### ASHRAE TC 9.9 — The Climate Rulebook
**ASHRAE** เป็นคนกำหนด **อุณหภูมิและความชื้น (temperature & humidity)** ที่ Server ต้องการ โดยแบ่งเป็น 2 ช่วง:
- **Recommended range** — ช่วงที่แนะนำ (จุดที่ Server ทำงานได้ดีและเสถียรที่สุด)
- **Allowable range** — ช่วงที่พอรับได้ (ขีดจำกัดสูงสุดก่อนที่อุปกรณ์จะเริ่มมีปัญหา)

#### NFPA — Fire Safety
**NFPA** (National Fire Protection Association) เป็นคนออกกฎเรื่องอัคคีภัย:
| Standard | สิ่งที่ควบคุม |
|----------|---------------|
| NFPA 75 | การปกป้องอุปกรณ์ IT จากไฟ |
| NFPA 76 | การปกป้องอาคารกลุ่ม Telecommunications |
| NFPA 70 (NEC) | ความปลอดภัยของ Electrical wiring (การเดินสายไฟ) |
| NFPA 13 | การออกแบบระบบ Sprinkler |

## 3. The Tier System

Data Center แต่ละที่สร้างมาไม่เหมือนกัน บางที่ก็ทั่วไป บางที่ก็ถึกทนระดับรอดจากภัยพิบัติใหญ่ ๆ ได้ **Uptime Institute** เลยสร้างระบบที่เรียกว่า **Tiers** เพื่อจัดระดับความน่าเชื่อถือ

### เข้าใจระบบ Tiers

| Tier | ชื่อเรียก | Uptime | Downtime ต่อปี | ความหมาย |
|------|------|--------|-------------------|---------------|
| **I** | Basic | 99.671% | ~28.8 ชั่วโมง | มี Power และ Cooling แค่ path เดียว (ไม่มี backup) ถ้ามีอะไรพัง ระบบล่มทั้งยวง |
| **II** | Redundant Components | 99.741% | ~22.7 ชั่วโมง | เหมือน Tier I แต่มี **อุปกรณ์ backup** (เช่น มี UPS สำรอง) แต่ก็ยังมี path เดียวอยู่ดี |
| **III** | Concurrently Maintainable | 99.982% | ~1.6 ชั่วโมง | **มีหลาย paths** สำหรับไฟและแอร์ สามารถซ่อมหรือเปลี่ยนอุปกรณ์ได้ **โดยที่ระบบไม่ต้อง downtime เลย** |
| **IV** | Fault Tolerant | 99.995% | ~26 นาที | มีทุกอย่างของ Tier III แต่ระบบสามารถ **รับมือกับ failure ได้อัตโนมัติ** ถ้าชิ้นส่วนไหนพัง ระบบก็ยังทำงานต่อได้แบบไม่มีกระตุก |

### เปรียบเทียบง่าย ๆ
- **Tier I** = บ้านที่มีถนนเข้าบ้านเส้นเดียว ถ้าถนนปิด ก็เข้าบ้านไม่ได้
- **Tier II** = บ้านหลังเดิม ถนนเส้นเดิม แต่มีล้ออะไหล่ติดรถไว้
- **Tier III** = บ้านมี **ถนนสองเส้น** ถ้าเส้นนึงปิด ก็ใช้อีกเส้นได้ (สลับทางได้โดยไม่ต้องหยุดชะงัก)
- **Tier IV** = ถนนสองเส้น และรถ **สลับเส้นทางให้อัตโนมัติ** ถ้ารู้ว่าทางข้างหน้ามีปัญหา โดยที่คุณไม่รู้สึกตัวเลยด้วยซ้ำ

> **ประเด็นสำคัญ:** Tier สูง ๆ จะสร้างแพงมาก แต่ก็แลกมาด้วย Downtime ที่น้อยลง ธนาคารหรือโรงพยาบาลมักใช้ Tier III หรือ IV แต่ถ้าเป็นเว็บบล็อกบริษัทเล็ก ๆ Tier I ก็อาจจะเพียงพอ

### อะไรเป็นสาเหตุหลักของ Downtime?
| สาเหตุ | เปอร์เซ็นต์ |
|-------|-----------|
| Planned maintenance (การซ่อมบำรุงตามแผน) | 30% |
| Server software issues (ปัญหาซอฟต์แวร์ของฝั่ง Server) | 30% |
| Human error (คนทำพลาด) | 15% |
| Hardware failure (อุปกรณ์พัง) | 10% |
| Network software | 5% |
| Client software | 5% |
| Environmental (เช่น overheat) | 5% |

*(จะเห็นว่า 60% มาจากการทำ maintenance และเรื่องของซอฟต์แวร์ ไม่ใช่ตัวฮาร์ดแวร์พัง Data Center ที่ดีเลยต้องเน้นเรื่อง Process และ Management ให้ดีพอ ๆ กับสเปคของเครื่อง)*

### High Availability Design — 4 เสาหลัก
Data Center ถูกออกแบบมาเพื่อลด Downtime โดยใช้หลักการ 4 อย่าง:
- **Reliability** — อุปกรณ์ถูกสร้างมาให้ทนทาน ทำงานสม่ำเสมอ
- **Resiliency** — ระบบฟื้นตัวเร็วเวลามีอะไรพัง
- **Availability** — Service พร้อมใช้งานตลอดเวลา
- **Serviceability** — บำรุงรักษาหรือเปลี่ยนอะไหล่ได้ง่ายโดยไม่กระทบระบบรวม

## 4. The Physical Building

### ใน Data Center หน้าตาเป็นยังไง?
เดินเข้าไปข้างใน จะเห็นตู้เหล็กสูง ๆ (เรียกว่า **Racks**) เรียงกันเป็นแถว ๆ บนพื้นเรียบ ๆ แต่ละ Rack จะใส่ Server ซ้อนกันไว้หลายตัว มีสายเคเบิลโยงอยู่ด้านบนหรือใต้พื้นเต็มไปหมด

องค์ประกอบหลัก ๆ:
- **Server Racks / Cabinets** — โครงตู้เหล็กที่เอาไว้วาง Server 
- **Cable Trays & Ladder Racks** — รางด้านบนที่เอาไว้วางสาย Network และ Power ให้เป็นระเบียบ
- **Aisles** — ทางเดินระหว่าง Racks (ออกแบบมาให้รับลมแอร์โดยเฉพาะ)
- **Ramps** — ทางลาด ไว้เข็นของหนัก ๆ
- **Structural Floor** — พื้นคอนกรีตจริงของตึก ที่ต้องรับน้ำหนักอุปกรณ์เป็นตัน ๆ ได้

### The Raised Floor
Data Center ส่วนใหญ่จะมี **Raised Floor** (ยกพื้นสูงขึ้นมาจากพื้นคอนกรีตจริงประมาณ 30-90 ซม.)
ช่องว่างใต้พื้นนี้เรียกว่า **Plenum** ซึ่งมีประโยชน์มาก:
1. **Cold air delivery** — ระบบ Cooling จะเป่าลมเย็นลงไปใต้พื้น แล้วให้ลมดันขึ้นมาตรง **perforated tiles** (กระเบื้องเจาะรู) ตรงหน้า Server พอดี
2. **Cable routing** — เอาไว้ซ่อนสายไฟและสาย Network ให้เป็นระเบียบ
3. **Grounding** — โครงเหล็กใต้พื้นช่วยทำหน้าที่เป็นสายดิน ป้องกันเรื่องไฟฟ้าสถิต

## 5. Power

### ทำไม Power ถึงสำคัญที่สุด?
ถ้าไฟดับ **ทุกอย่างจบ** Server ดับ เว็บล่ม Transaction พัง Data Center เลยต้องมี **ระบบกันไฟดับหลายชั้น (multiple layers of power protection)**

Flow ของระบบไฟจะเป็นแบบนี้:
```
Utility Power (ไฟบ้าน / ไฟหลวง)
    ↓
ATS/STS (สวิตช์สลับไฟ)
    ↓
UPS (เครื่องสำรองไฟ)
    ↓
PDUs (ตู้แบ่งไฟ)
    ↓
Power Panels
    ↓
Individual Servers
```

### อธิบายทีละ Layer
1. **Utility Power** — ไฟหลวง เป็นแหล่งพลังงานหลัก แต่ก็มีโอกาสไฟตกหรือดับได้
2. **ATS (Automatic Transfer Switch)** — ตัวจับว่าถ้าไฟหลวงดับ ให้สลับไปใช้ Generator ทันที / ส่วน **STS (Static Transfer Switch)** คือสลับไฟแบบเสี้ยววินาที (ไม่มีชิ้นส่วนขยับ) จน Server ไม่รู้ตัวเลยว่าไฟกระพริบ
3. **UPS (Uninterruptible Power Supply)** — แบตเตอรี่ลูกยักษ์ เอาไว้อุดรอยรั่วช่วงเสี้ยววินาทีตอนไฟตก ก่อนที่ Generator จะสตาร์ทติด (ช่วยพยุงระบบได้ราว 5-30 นาที)
4. **Power Generator** — เครื่องปั่นไฟ (มักใช้น้ำมันดีเซล) เป็น Backup engine ที่รันปั่นไฟให้ระบบอยู่ได้เป็นวัน ๆ
5. **PDUs (Power Distribution Units)** — ตัวรับไฟแรงสูงมาแบ่งเข้า Racks อารมณ์เหมือนปลั๊กพ่วงเวอร์ชันอุตสาหกรรม ที่ชาญฉลาดและคอยเช็คได้ด้วยว่าแต่ละ Rack กินไฟไปเท่าไหร่
6. **Power Panels** — จุดสุดท้ายที่จ่ายไฟเข้า Server ในแต่ละ Rack

### Types of Electrical Power
- **Single-Phase Power** — ไฟบ้านปกติ ใช้กับอุปกรณ์ที่กินไฟน้อย
- **Three-Phase Power** — ไฟ 3 เฟส พลังงานเยอะและมีประสิทธิภาพกว่า อุปกรณ์ใน Data Center แทบทั้งหมดใช้แบบนี้

## 6. Cooling

### ทำไมต้อง Cooling?
Server ปล่อยความร้อนเยอะมาก Rack นึงอาจปล่อยความร้อนเท่ากับเอาฮีตเตอร์ 10-20 ตัวมาเปิดพร้อมกัน ถ้าไม่ทำความเย็น:
- Server จะ **Overheat** แล้วดับตัวเอง
- Performance ตก เพราะ CPU ต้องลดความเร็วลงเพื่อคุมอุณหภูมิ
- ฮาร์ดแวร์อายุสั้นลง
- Service ล่ม

### Server ต้องการอุณหภูมิเท่าไหร่?
ASHRAE ระบุไว้ว่า:
- **Temperature**: 18°C - 27°C
- **Relative Humidity (ความชื้นสัมพัทธ์)**: 40% - 60%
> ถ้าอากาศแห้งไป → ไฟฟ้าสถิตสะสมแล้วช็อตได้ / ถ้าชื้นไป → หยดน้ำเกาะบอร์ดแล้วช็อตพัง

### PAC (Precision Air Conditioning)
Data Center ไม่ใช้แอร์บ้าน แต่ใช้ **PAC** แอร์บ้านทั่วไปเน้นดึงความชื้นออกเพื่อให้คนสบายตัว แต่ PAC เน้นดึงความร้อน (heat) ล้วน ๆ เพื่อให้ Server สบายตัว แถมยังคุม Humidity และรักษา Air pressure ภายในห้องให้เป็นบวก (กันฝุ่นเข้า) ได้แม่นยำมาก

### ระบบ Cooling หลัก ๆ 2 แบบ
1. **CRAC (Computer Room Air Conditioner)** — เหมือนแอร์บ้านแต่สเกลใหญ่ ใช้ **Refrigerant (น้ำยาแอร์)** ดึงความร้อนไปทิ้งข้างนอก
2. **CRAH (Computer Room Air Handler)** — ใช้ **Chilled water (น้ำเย็น)** จากส่วนกลาง (Chiller plant) วิ่งเข้ามาทำความเย็นให้ลม (เหมาะกับ Data Center สเกลใหญ่ ประหยัดไฟกว่า)

> [YouTube: CRAC vs CRAH](https://youtu.be/5iTHrqeVAO8?si=aR76_sNRVdqdMv8p)

### ทิศทางลม (Hot Aisle / Cold Aisle)
เพื่อให้ลมเย็นทำงานคุ้มค่าที่สุด เขาจะจัด Rack เป็นแถว ๆ หันหน้าไปทางเดียวกัน:
- **หน้า Server** (ฝั่งที่ดูดลมเย็นเข้า) จะหันเข้าหา **Cold Aisle** (ลมเย็นจะดันขึ้นมาจาก perforated tiles)
- **หลัง Server** (ฝั่งที่เป่าลมร้อนออก) จะหันเข้าหา **Hot Aisle** (พ่นลมร้อนออกไปให้แอร์ดูดกลับ)

#### Containment (การกั้นห้องลม)
ป้องกันไม่ให้ลมร้อนกับลมเย็นผสมกัน (ลดการสูญเสียพลังงาน)
- **HAC (Hot Aisle Containment)** — กั้นห้องฝั่งทางเดินลมร้อน
- **CAC (Cold Aisle Containment)** — กั้นห้องฝั่งทางเดินลมเย็น

### PUE — ค่าชี้วัดความประหยัดไฟ
**PUE (Power Usage Effectiveness)** = (พลังงานทั้งหมดที่ตึกใช้) ÷ (พลังงานที่ส่งให้ IT Equipment ล้วน ๆ)
- ค่า **1.0** คือ Perfect สุด ๆ (พลังงานทั้งหมดลง Server ไม่มีเสียเปล่าไปกับแอร์เลย)
- ค่า **2.0** คือครึ่งนึงเข้า Server อีกครึ่งหายไปกับแอร์และไฟแสงสว่าง
- Data Center เทพ ๆ สมัยนี้มักทำ PUE ได้ที่ **1.2 - 1.4**

## 7. Fire Protection

ถ้าใช้ Sprinkler ฉีดน้ำลง Server ข้าวของก็พังพอ ๆ กับโดนไฟไหม้ Data Center เลยต้องใช้:
1. **Detection:** ใช้ **VESDA (Very Early Smoke Detection Apparatus)** เป็นตัวสูดอากาศไปเข้าเครื่องวิเคราะห์เลเซอร์ตลอดเวลา สามารถตรวจจับควันได้ตั้งแต่เพิ่งเริ่มคุกรุ่น (ก่อนมนุษย์จะดมกลิ่นได้ หรือก่อนมีเปลวไฟด้วยซ้ำ)
2. **Suppression:** ใช้ระบบ **Clean Agent** (ก๊าซดับเพลิง) เช่น **FM-200** หรือ **Novec 1230** ที่พอพ่นปุ๊บจะไปลดอุณหภูมิจนไฟดับ โดยที่ไม่ทิ้งคราบและไม่ทำให้อุปกรณ์อิเล็กทรอนิกส์ช็อต

---

## 8. Security

ต้องป้องกันแบบเป็น Layers เหมือนหัวหอม (คนร้ายต้องเจาะเข้ามาทีละชั้น):
1. **Perimeter security** — รั้ว, รปภ., ป้อมยามรอบตึก
2. **Building access** — Access control (บัตรคีย์การ์ด, PIN)
3. **Biometric** — สแกนลายนิ้วมือ, สแกนม่านตา, สแกนใบหน้า ก่อนเข้าโซนสำคัญ
4. **Mantrap entry** — ประตู 2 ชั้น (ประตูบานแรกต้องปิดสนิท บานที่สองถึงจะเปิดได้) ช่วยป้องกันคนเนียนเดินตามหลัง (Tailgating)
5. **CCTV** — กล้องวงจรปิดครอบคลุมทุกมุม 24/7
6. **Visitor management** — คนนอกทุกคนต้องแลกบัตร มีคนประกบ และโดนจับตาตลอด

---

## 9. Monitoring

- **NOC (Network Operations Center)** — ห้อง Command center ที่มีจอใหญ่ ๆ แปะเต็มกำแพง มีทีมงานนั่งเฝ้าระบบ 24/7 ดูทั้ง Power, Cooling, Network และ Security มีอะไรผิดปกตินิดเดียว ทีมนี้รู้ทันที
- **DCIM (Data Center Infrastructure Management)** — ซอฟต์แวร์ Dashboard รวมศูนย์ ไว้ให้ผู้จัดการมอนิเตอร์ภาพรวมของตึกแบบครบจบในที่เดียว (ดูว่าตู้ไหนร้อน สายเส้นไหนหลุด กินไฟไปเท่าไหร่)

## 10. Disaster Recovery (DR)

ต่อให้ตึกดีแค่ไหน ภัยพิบัติ (น้ำท่วม, ไฟไหม้, แผ่นดินไหว) ก็เกิดได้ **Disaster Recovery (DR)** คือแผนว่าถ้า **Primary site (Data Center หลัก)** พัง เราจะมี **DR site (ศูนย์สำรอง)** มารับช่วงต่อยังไง

### ต้องตั้งห่างกันแค่ไหน? (Geographic Separation)
- **ห่างไปเลย 120-160 กม. (แนะนำ)** — ไกลพอที่จะไม่โดนพายุหรือภัยพิบัติวงกว้างลูกเดียวกัน แต่ก็ไม่ไกลเกินไปจน Network delay (Latency) พุ่งสูงจน Sync ข้อมูลช้าเกินรับได้

### ประเภทของ DR Site
- **Cold Site** — ตึกเปล่า ๆ มีระบบไฟกับแอร์รอไว้ แต่ไม่มีของ (ต้องขน Server ไปลงเอง) ใช้เวลากู้คืนเป็นวัน ๆ หรือเป็นสัปดาห์ (ถูกสุด)
- **Warm Site** — มีฮาร์ดแวร์รอไว้บ้างแล้ว แต่ต้องมาลงข้อมูล / Config เอาเอง (กู้คืนระดับวัน)
- **Hot Site** — Mirror จากศูนย์หลักทุกประการ รันสแตนด์บายรอไว้ตลอดเวลา สลับมาใช้งานได้ทันทีในระดับนาที (แต่แพงหูฉี่)

### 2 ตัวเลขสำคัญของการทำ DR
1. **RTO (Recovery Time Objective)** — *"เรายอมล่มได้นานสุดแค่ไหน?"* (How fast must we recover?) เช่น RTO=2 ชม. แปลว่าระบบต้องกลับมาใช้งานได้ใน 2 ชม. หลังเกิดเหตุ
2. **RPO (Recovery Point Objective)** — *"เรายอมให้ข้อมูลย้อนหลังหายได้มากสุดแค่ไหน?"* (How much data can we lose?) เช่น RPO=15 นาที แปลว่าอย่างน้อยต้องมี Backup ข้อมูลล่าสุดของ 15 นาทีที่แล้ว งานที่ทำในช่วง 15 นาทีสุดท้ายอาจจะต้องยอมทิ้ง

### การทำ Data Replication
- **Offline Replication** — ขนฮาร์ดดิสก์รอบดึกไปส่งที่ศูนย์สำรอง (ช้า ข้อมูลล้าหลัง)
- **Online Replication (แบบ Synchronous)** — เขียนข้อมูลลง 2 ที่พร้อม ๆ กัน (RPO เป็น 0) ข้อมูลไม่หายเลย แต่ต้องอยู่ไม่ไกลกันมากและเน็ตต้องแรงจัด
- **Online Replication (แบบ Asynchronous)** — เขียนลงที่หลักก่อน แล้วค่อยส่งอัปเดตไปที่ DR Site ทีหลัง (ยอมให้ข้อมูลดีเลย์นิดหน่อยช่วงกำลังวิ่งไปตามสาย แต่ข้อดีคือตั้งศูนย์สำรองห่างกันไกล ๆ ข้ามภาค ข้ามประเทศได้)

## สรุปแบบเร็ว  ๆ

| หัวข้อ | ประเด็นหลัก |
|-------|-------------|
| **Data Center** | ตึกเฉพาะทางที่คุมสภาพแวดล้อมให้ Server รันได้ 24/7 (มี Power, Cooling, Security พร้อม) |
| **Standards** | TIA-942 (เรื่องดีไซน์), ASHRAE (เรื่องแอร์/อุณหภูมิ), NFPA (เรื่องป้องกันไฟ) |
| **Tiers** | Tier I (เบสิค) จนถึง Tier IV (รอดตายอัตโนมัติ) — Tier ยิ่งสูง ยิ่งนิ่ง ยิ่งแพง |
| **Power** | 3 Layers: Utility Power (ไฟหลวง) → UPS (แบตเตอรี่อุดรอยรั่ว) → Generator (ปั่นไฟยาว ๆ) |
| **Cooling** | ระบบ CRAC/CRAH + ทิศทางลม Hot/Cold aisle คุมให้อยู่ที่ 18-27°C |
| **Fire Protection** | VESDA จับควันได้ไวมาก + Clean agents (FM-200, Novec) ดับไฟแบบที่อุปกรณ์ไม่เปียกชื้น |
| **Security** | Layers ป้องกัน: คีย์การ์ด → Biometrics → Mantrap → CCTV |
| **Monitoring** | ทีม NOC + ระบบซอฟต์แวร์ DCIM นั่งเฝ้าระบบตลอด 24/7 |
| **Disaster Recovery** | มี DR site (Cold/Warm/Hot) รอรับช่วงต่อ โดยต้องโฟกัสที่ RTO (เวลาล่ม) และ RPO (ข้อมูลหาย) |
| **PUE** | (ไฟที่ใช้ทั้งหมด) ÷ (ไฟที่เข้า Server) — เข้าใกล้ 1.0 คือดีเยี่ยม |
